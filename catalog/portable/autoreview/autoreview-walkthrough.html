2. Skill Purpose

The skill defines autoreview as a structured code review closeout check. It is meant to be used before commit, before ship, after non-trivial code edits, or when explicitly requested as a Codex/Claude/second-model review.

The skill draws a boundary early: this is code review, not Guardian auto_review approval routing. In other words, it is not a policy gate that approves actions. It is a reviewer helper that returns findings the main agent must still verify.

Default engine

The skill says Codex review is the default when no engine is set and should remain the normal final closeout engine. The script mirrors that with:

parser.add_argument("--engine", choices=ENGINES, default=os.environ.get("AUTOREVIEW_ENGINE", "codex"))

So the order of precedence is: explicit --engine, then AUTOREVIEW_ENGINE, then codex.

3. Agent Contract

The SKILL.md contract is the human side of the system. It prevents the agent from treating another model as an unquestionable oracle. The major rules are:

  • Review output is advisory.
  • Every finding must be verified in the real code path.
  • Dependency behavior should be checked in docs, source, or types when relevant.
  • Speculative risks, unrealistic edge cases, broad rewrites, and over-complicated fixes should be rejected.
  • If a real finding reveals a repeated bug class, inspect sibling instances in the current PR scope.
  • If a fix changes code, rerun focused tests and rerun autoreview.
  • Do not switch the requested engine/model merely because capacity is annoying. Retry the same engine/model a few times instead.
  • Do not push just to review.
  • Stop once the helper exits cleanly with no accepted/actionable findings.
The core idea:

The script can automate bundling, prompting, validation, and exit codes. It cannot decide whether a finding is worth changing. That remains the main agent's responsibility.

4. Target Selection

The skill gives user-facing target guidance, while choose_target() enforces it in code. The target decides what git data gets bundled for review.

Mode Use Case Bundle Source
local Dirty staged, unstaged, or untracked work. git status, staged diff, unstaged diff, and untracked file text.
uncommitted Alias for local. Same as local.
branch PR or branch diff. git diff base...HEAD.
commit One already-committed change. git show for the requested commit ref.
auto Default behavior. Dirty local first, otherwise branch on non-main, otherwise error on clean main.

The code path

def choose_target(repo: Path, mode: str, base_ref: str | None) -> tuple[str, str | None]:
    mode = "local" if mode == "uncommitted" else mode
    branch = current_branch(repo)
    if mode == "local" or (mode == "auto" and is_dirty(repo)):
        return "local", None
    if mode == "commit":
        return "commit", None
    if mode == "branch" or (mode == "auto" and branch != "main"):
        return "branch", base_ref or detect_pr_base(repo) or "origin/main"
    raise SystemExit("no review target: clean main checkout and no forced mode")

The interesting bit is the branch fallback: if --base is not supplied, the script tries gh pr view --json baseRefName --jq .baseRefName. If that works, it reviews against the real PR base. If not, it falls back to origin/main.

5. Script Overview

scripts/autoreview is one Python executable with a straightforward vertical flow. It does not import project-local modules, which makes it easy to install with a skill and run from arbitrary repos.

1. ParseRead CLI flags and environment defaults.
2. DetectFind git repo, branch, dirty state, and review target.
3. BundleCollect git diff/status/show text into one review bundle.
4. PromptEmbed schema, rules, optional context, and bundle.
5. InvokeRun Codex, Claude, Droid, Copilot, or a panel.
6. ExtractRecover structured JSON from engine output.
7. ValidateEnforce schema and changed-file scope.
8. ExitPrint report and return 0/1 based on findings/tests.

There is one mild architectural smell worth noting: everything lives in a single file. For a portable helper, that is reasonable. For a larger maintained tool, the natural split would be git_bundle.py, engines.py, schema.py, and panel.py. The current single-file design favors easy copying and installation.

6. Structured Schema

The script starts by defining supported engines and thinking levels:

ENGINES = ("codex", "claude", "droid", "copilot")
THINKING_LEVELS_BY_ENGINE = {
    "codex": {"low", "medium", "high", "xhigh"},
    "claude": {"low", "medium", "high", "xhigh", "max"},
    "droid": set(),
    "copilot": set(),
}

Then it defines SCHEMA, the exact JSON object expected from the reviewer. Required top-level fields:

  • findings: an array of structured findings.
  • overall_correctness: either patch is correct or patch is incorrect.
  • overall_explanation: non-empty string, max 3000 chars.
  • overall_confidence: numeric confidence from 0 to 1.

Each finding must contain:

  • title: 1 to 140 chars.
  • body: 1 to 2000 chars.
  • priority: P0, P1, P2, or P3.
  • confidence: number from 0 to 1.
  • category: bug, security, regression, test_gap, or maintainability.
  • code_location: object with file_path and 1-based line.

The schema is sent to engines where possible and then enforced again locally. This double-check is important because model CLIs can wrap, stringify, stream, or otherwise reshape their output.

7. Subprocess Layer

run()

run() is the small synchronous subprocess wrapper. It always captures stdout/stderr as text. When check=True, it exits the whole helper with a useful diagnostic if the subprocess fails.

run_with_heartbeat()

Model calls can be slow, so run_with_heartbeat() avoids looking hung. Without streaming, it calls communicate() with a timeout. Every timeout prints:

review still running: <engine> elapsed=<seconds>s pid=<pid>

This directly supports the skill instruction that heartbeats are healthy progress and should not be killed just because the model is quiet for a few minutes.

run_with_stream()

When --stream-engine-output is set, stdout and stderr are consumed by separate daemon threads. Lines go into a queue so the main thread can multiplex output, preserve full buffers for validation, and display filtered live status.

8. Repo And Command Safety

The helper is reviewing possibly untrusted code, so command lookup is unusually cautious.

Repository root

repo_root() walks upward looking for .git, finds a trusted git binary, then runs git rev-parse --show-toplevel. If not inside a git repo, it exits.

Executable resolution

find_command() has several safety rules:

  • If the command has a directory component, an absolute path is used as-is and a relative path resolves from the reviewed repo root.
  • For bare commands like git, codex, or claude, only absolute PATH entries are considered.
  • Empty PATH entries, ., and relative PATH entries are ignored.
  • Any executable found under the reviewed repo root is rejected.
  • On Windows, PATHEXT candidates are considered for extensionless names.

This defends against a repo slipping a fake git, claude, or codex earlier in PATH and getting the review helper to execute it.

9. Bundle Building

The helper does not simply say "review this repo". It constructs a frozen text bundle that represents the review target.

local_bundle(repo)

Local mode includes:

  • git status --short
  • git diff --cached --stat
  • git diff --cached --patch --find-renames
  • git diff --stat
  • git diff --patch --find-renames
  • Full contents of untracked files from git ls-files --others --exclude-standard

Binary untracked files are omitted. Long text files are truncated by bounded().

branch_bundle(repo, base_ref)

Branch mode does a quiet best-effort git fetch origin, then includes the diff stat and patch for base_ref...HEAD. The triple-dot syntax matters: it compares the branch against the merge base with the base ref, which is what PR review usually wants.

commit_bundle(repo, commit_ref)

Commit mode includes git show --stat --format=fuller and git show --patch --find-renames --format=fuller. This is the right mode for reviewing one landed or already-pushed commit.

review_paths()

This function computes the set of changed file paths separately from the textual bundle. Validation later uses this set to discard findings outside the reviewed change.

10. Prompt Construction

build_prompt() creates the reviewer instruction. Its hard rules are the core behavioral contract passed to the engine:

  • Return exactly one JSON object and nothing else.
  • Match the schema exactly.
  • Do not modify files.
  • Do not invoke nested reviewers or review tools.
  • Use only read-only inspection commands if shell tools exist.
  • Report only actionable defects introduced or exposed by the change.
  • Include concrete security issues, but do not flag legitimate shell/filesystem/auth/sensitive-data functionality merely because it looks risky.
  • Use the smallest file/line location that demonstrates the issue.
  • Return an empty findings array and mark the patch correct when there are no actionable findings.

The prompt then records the review target, repo path, optional extra prompt text, optional datasets, and the full change bundle. --prompt, --prompt-file, and --dataset are first-class ways to feed additional context without changing the bundle builder.

11. Engine Adapters

Codex

run_codex() is the default path. It writes the JSON schema to a temp file and creates another temp file for Codex's final output. The command uses:

  • --ask-for-approval never
  • --search when web search is enabled
  • --model <model> when supplied
  • -c model_reasoning_effort="<thinking>" for Codex thinking levels
  • exec --ephemeral
  • -C <repo>
  • -s read-only
  • --output-schema <schema_path>
  • --output-last-message <output_path>
  • - to read the prompt from stdin

Codex rejects --no-tools because this review mode depends on read-only inspection tools. After the process exits, temp files are cleaned up and the final message file is preferred over stdout.

Claude

run_claude() calls Claude with print mode, no session persistence, JSON or stream-json output, and the JSON schema. When tools are enabled, allowed tools default to Read,Grep,Glob,WebSearch,WebFetch. If --no-web-search is set, web tools are removed from that list. Claude maps thinking to --effort.

Droid

run_droid() writes the prompt to a temp file and invokes droid exec --cwd <repo> --output-format json -f <prompt_path>. It supports --model and --disabled-tools *, but rejects --thinking.

Copilot

run_copilot() writes prompt.txt into a private tempdir with mode 0600. It asks Copilot to read that file and return only JSON. It disables builtin MCPs, avoids asking the user, and explicitly allows read_agent, rg, view, and web_fetch. It rejects --no-tools because the prompt is intentionally loaded through a read tool rather than exposed in argv.

12. Streaming Output Filters

CodexStreamDisplay and ClaudeStreamDisplay make live streaming useful without flooding the terminal with tool chatter.

Codex stream display

  • Shows codex thread: <id> on thread start.
  • Shows codex turn started.
  • Shows agent messages.
  • Shows usage on turn completion.
  • Counts hidden tool/status events and occasionally prints compact activity summaries.

Claude stream display

  • Shows claude turn started on the first system event.
  • Extracts assistant text chunks.
  • Shows token and cost usage on result events.
  • Compacts hidden events like the Codex display.

13. JSON Extraction

Real model CLIs do not always return exactly the same wrapper shape, so extract_json() is strict about the final report but flexible about how to find it.

Accepted input shapes include:

  • A direct JSON object containing findings.
  • A JSON object with structured_output.
  • A JSON object with string result containing nested JSON.
  • Markdown-fenced JSON.
  • JSONL streams where text, content, result, or structured output events contain the report.
  • Nested JSON strings, recursively parsed when useful.

If no valid report can be found, the helper exits with a clear message and a prefix of the unexpected output.

14. Validation And Scope Filtering

validate_report() is one of the most important functions. It revalidates everything the schema declared, but in normal Python rather than relying on a model CLI to enforce it.

Top-level validation

  • No unexpected top-level keys.
  • All required top-level keys present.
  • findings is an array.
  • overall_correctness is one of the two allowed strings.
  • overall_explanation is non-empty and not over 3000 chars.
  • overall_confidence is a non-boolean number from 0 to 1.

Finding validation

  • Each finding is an object.
  • No unexpected finding keys.
  • All required finding keys present.
  • title and body are non-empty and bounded.
  • priority and category are valid enum values.
  • confidence is a non-boolean number from 0 to 1.
  • code_location.file_path is relative and cannot contain ...
  • code_location.line is a positive integer.

Scope filtering

If a finding points at a file outside changed_paths, it is ignored. The helper prints a warning to stderr with the title, location, and truncated body. Then it removes those findings from the report.

If all findings were out-of-scope and the engine said the patch was incorrect, the helper changes the verdict to patch is correct and appends a note explaining that out-of-scope findings were ignored. This is a strong design choice: autoreview is reviewing the patch, not auditing the whole repository.

Required finding text

--require-finding searches across kept findings. If a required substring is missing, validation fails. The smoke harness uses this to ensure the malicious fixture catches a command-related issue.

15. Review Panels

Panels are opt-in through --panel or --reviewers. The skill explicitly discourages routine panels because the normal workflow is one structured review, verified by the main agent.

Reviewer specs

Supported forms include:

--reviewers codex,claude
--panel
--reviewers codex:gpt-5.1:high,claude:sonnet:max
--model codex=gpt-5.1 --thinking claude=max

reviewer_args() parses global and per-engine model/thinking values, validates thinking levels against engine support, clones the main args object for each reviewer, and prevents duplicate reviewers.

Concurrent execution

run_panel() submits each reviewer to a ThreadPoolExecutor. Failures normally fail the whole panel unless --allow-partial-panel is set. If partial panels are allowed, the helper prints failed reviewer labels and continues with successful reports.

Merging reports

merge_panel_reports() deduplicates by file path, line, category, and normalized title. It prefixes each finding body with the reviewer label and marks the panel incorrect if any merged finding exists or any reviewer returned an incorrect verdict.

16. Parallel Tests

--parallel-tests lets the helper run a focused test command while the model is reviewing. This is useful because review can take a while, and tests are often independent once formatting is done.

start_parallel_tests() supports these shell modes:

  • default: Python shell=True platform behavior.
  • cmd: same as default in the current implementation.
  • powershell: uses powershell -NoProfile -ExecutionPolicy Bypass -Command.
  • pwsh: uses pwsh -NoProfile -Command.

finish_parallel_tests() always waits and prints tests exit: <code> after <seconds>s. The main flow calls it in a finally block, so tests are not orphaned just because review extraction or validation fails.

17. Main Flow

The main() function is the whole helper in miniature:

  1. Parse CLI args.
  2. Build one or more reviewer argument objects.
  3. Find the git repository root.
  4. Choose the target and target ref.
  5. Print target metadata: target, branch, engine/reviewers, tools, web search, and ref.
  6. Return immediately for --dry-run.
  7. Build the correct bundle for local, branch, or commit mode.
  8. Build the prompt with optional prompt/dataset context.
  9. Compute changed paths.
  10. Print prompt size as bundle: <chars> chars.
  11. Start parallel tests if requested.
  12. Run one reviewer or a panel.
  13. Write --json-output if requested.
  14. Print the human report, optionally teeing it to --output.
  15. Wait for parallel tests.
  16. Return an exit code based on test status, findings, and verdict.

Exit status semantics

Condition Exit
Tests failed 1
--expect-findings and findings exist 0
--expect-findings and no findings exist 1
Findings exist 1
Overall verdict is incorrect 1
No findings, correct verdict, tests passed 0

18. Smoke Harness

test-review-harness.py is not a unit test suite for the helper internals. It is an acceptance smoke harness for review quality and calibration.

Fixtures

The harness has two fixture families:

Fixture Initial Version Changed Version Expected Result
malicious A simple upload path sanitizer and safe public user projection. Unsafe upload path interpolation, shell command construction with rm -rf, and password exposure. Findings, including text containing command.
benign A safe path helper and safe public user projection. Security-sensitive but safe use of paths, execFile, repo validation, and owner-gated account settings. Clean review.

Harness flow

  1. Parse --fixture and repeatable --engine. Defaults are codex and claude.
  2. Create a temp directory with prefix autoreview-fixture..
  3. Run git init and configure fixture author identity.
  4. Write the initial app.js.
  5. Commit it as initial safe version.
  6. Overwrite app.js with the changed fixture, leaving dirty local work.
  7. Run scripts/autoreview --mode local --engine <engine> --prompt <fixture prompt>.
  8. For malicious fixtures, append --require-finding command --expect-findings.
  9. Clean up the temporary repo, making files writable if needed.

Wrappers

The Bash wrapper finds python3 first, then python, then exits 127. The PowerShell wrapper accepts -Fixture, repeatable -Engine, and -Help, then prefers the Windows py -3 launcher before python.

19. Mental Model

The cleanest way to think about autoreview is as two cooperating layers:

The skill file is the judgment layer.

It tells the agent when to run review, how skeptical to be, when to rerun, when to stop, and how to summarize the final result.

The script is the mechanics layer.

It turns a git change into a frozen prompt bundle, invokes a selected model CLI under read-oriented constraints, normalizes output into a validated report, filters unrelated findings, optionally runs tests, and returns an exit code automation can trust.

The most valuable design choices are the structured schema, changed-file scope filtering, defensive executable resolution, read-only reviewer posture, optional but not default panels, and explicit rerun discipline after review-triggered fixes.
